Rescue & Emergency Apps
Closing the Loop on DRONE_A

Install both Android apps, then run the full system end to end: a citizen's phone detects your aux module over Bluetooth, sends an SOS, and a rescuer claims it with their PIN identity. No drone and no second module needed.

Needs 1 Android phone (2 is better) Node DRONE_A, already live New installs none on Windows
aux module BLE→ emergency app notified→ joins RESCUE_A→ SOS→ rescue app claims it
The APKs are already built for you. Download them straight to the phone from the phone-apps-v1 release. Nothing to install on the Windows laptop. You will need the personnel ID + PIN you issued from the GCC, and the fleet CA text.
Get the latest of everything before you start. The 15 July build fixes three field-reported bugs: the HQ Uplink tab logging a rescuer out, the emergency app never seeing the module over Bluetooth, and the SOS button staying greyed out after joining Wi-Fi manually.
  • Re-download both APKs from the release (they replace your installed copies).
  • Re-flash the aux module with the current firmware — the Bluetooth fix is in the firmware, so an un-reflashed module still will not be discovered. Use the aux runbook's pio run -t upload step.
Both apps bind themselves to the drone's Wi-Fi, so you can leave mobile data on.

00

Get the fleet CA text onto the phone

The rescue app verifies the node against the fleet CA and refuses imposters, so it needs the certificate text. Do this while you still have internet, because RESCUE_A has none.

Print the certificate on the Pi and copy it:

Pi shell (SSH)
$ cat ~/rescue-mesh/deploy/secrets/fleet_ca.crt
Copy the whole block, including both marker lines
-----BEGIN CERTIFICATE-----
MIIB...
...
-----END CERTIFICATE-----

Get that text to the phone however is easiest: email it to yourself, or paste it into a notes app that syncs. You will paste it into the rescue app in Stage 2.

Also have ready the personnel ID and PIN you issued in the GCC (e.g. R-014 + a 6-digit PIN).

01

Install both apps

On the phone's browser, open the release page and download both APKs:

Release
github.com/dronecomnet-fyp/drone-network-system
                    /releases/tag/phone-apps-v1

Download rescue-app-v2.apk and emergency-app-v0.1.apk.

Android will ask permission to install from your browser the first time: allow "Install unknown apps" for that browser, then tap each downloaded APK to install.

Two phones is better than one. If you have a second Android phone, put the rescue app on one and the emergency app on the other. That way you can watch the SOS arrive on the rescuer's phone in real time. With one phone, just switch between the two apps.

Part A — Rescue app (file 05)

02

Rescue app: log in with the PIN

Join the phone to the RESCUE_A WiFi (open, no password), then open the rescue app. It starts on the login screen.

Before logging in, tap "Connection settings / break-glass admin key" at the bottom of the login screen and set:

Settings
Backend URL
leave as https://10.42.0.1:8443
Fleet CA certificate (PEM)
paste the whole certificate text from Stage 0
Admin API key
leave EMPTY (you are logging in with a PIN, not the break-glass key)
Save Settings
tap it, then go back

Now log in with the personnel ID and PIN from the GCC.

Leave "Accept ANY certificate" OFF. With the CA pasted, the app is doing real verification, which is the point: an imposter WiFi pretending to be RESCUE_A would be rejected.
03

Gate — claim a request with your identity

Once logged in you land on Victim Requests. The test message you sent through the captive portal should be listed.

  1. Tap CLAIM REQUEST on it and confirm.
  2. The card should flip to a green "CLAIMED by R-014" badge (your ID, not an anonymous badge).
  3. Check the Announcements tab: the broadcast you published from the GCC should be there.
Stop here

Confirm three things: you logged in with the PIN (no API key typed anywhere), the claim shows your personnel ID, and the GCC's announcement appears. That proves the full authenticated plane from the operator's desk to a rescuer's phone. Bonus: open the GCC's Live Feed and watch the same message now show as claimed by you.

Part B — Emergency app (file 06)

04

Emergency app: work through the permissions

Open the emergency app. It walks you through four permissions one at a time, each with a plain reason. Grant all four:

  • Notifications — how you get told a drone is near.
  • Location — the local log stored on the phone.
  • Background location ("Allow all the time") — so logging continues with the app closed. Android may make you pick this in a separate settings screen.
  • Nearby devices — the Bluetooth scan for the drone.

Then open Settings → Log a point now once, and check Your data: you should see one stored point, marked "on phone only".

That "on phone only" label is the privacy story in one line: nothing has left the device yet, and nothing will until you reach a drone or press SOS.
05

Arm the Bluetooth watch

First, disconnect the phone from RESCUE_A WiFi (forget it or turn WiFi off) so you can prove discovery works from scratch, the way a real citizen would experience it.

Make sure the aux module is powered and in NORMAL mode (plugged into the Pi and being pinged — that is its normal running state; it advertises over BLE only in normal mode).

On the emergency app's home screen, flip Watch armed on. A persistent "Emergency watch active" notification appears — that is the foreground service Android requires for continuous scanning.

Now lock the phone and walk it near the aux module.

06

Gate — the drone-found alert

Expected on the locked phone
Rescue drone nearby
Tap to connect and send your location. Network: RESCUE_A

Tapping it opens the Drone found screen showing the node and the SSID RESCUE_A — both read straight out of the module's Bluetooth advertisement, not hardcoded.

Stop here

Confirm the notification arrives on a locked phone and names RESCUE_A. This is the honest limit of what any mobile OS allows: an app cannot open itself from a background scan, so detection produces a high-priority alert and the user taps it. If no notification appears, check the module is in normal mode (not fallback) — see troubleshooting.

07

Join, upload, and send the SOS

From the Drone found screen:

  1. Tap Open Wi-Fi settings. The SSID is copied for you; join RESCUE_A.
  2. Come back to the app. Within a few seconds the home screen updates on its own and the SOS button turns red — tap it. (You do not have to come through the notification: joining RESCUE_A any time, via the "Connect to a drone manually" button on Home or the phone's Wi-Fi settings, lights up SOS the same way.)
  3. The Connected screen opens and immediately uploads your stored location points: "Sent 1 saved location point(s) to the rescue team."
  4. Type a short SOS message (e.g. "two adults, one injured, on the roof") and tap SEND SOS NOW.
Check Your data again afterwards: the point that said "on phone only" now says "sent to rescuers". The app is honest with the user about exactly what left the device and when.
08

Gate — close the loop

Now switch to the rescue app (or the second phone, still logged in as your rescuer).

Expected in Victim Requests
[SOS] two adults, one injured, on the roof
NEW  |  via DRONE_A  |  GPS 6.92710, 79.86120

The SOS arrived as a normal rescue request, with the citizen's GPS attached. Claim it — it should flip to "CLAIMED by R-014".

In the GCC on Windows, the same SOS appears in Live Feed, and the uploaded location points appear on the Map as checkin dots.

Stop here — this is the whole system

You have just run the complete chain on real hardware: a Bluetooth beacon from your soldered module reached a citizen's locked phone, the citizen joined the drone's network and sent an SOS with their location history, and a rescuer authenticated with a PIN issued from HQ claimed it under their own identity, all with no internet anywhere. That is the demo. Tell me it worked and I will help you capture it for the report.

Troubleshooting

SymptomTry this
Either app: "Cannot reach the drone ... (Connection failed)"Make sure you have the current APKs (re-download from the release; builds before 14 Jul had no network permission and could never connect). The apps now bind themselves to the drone's Wi-Fi, so mobile data can stay on. If it persists, confirm in the phone's browser that http://10.42.0.1/ loads the portal.
Rescue app: "TLS rejected" (a certificate error, not a connection one)The pasted CA is wrong or truncated. Re-copy the whole block including the BEGIN and END lines.
Rescue app: login failsCheck the personnel ID format (e.g. R-014) and the PIN. Five wrong PINs trigger a rate limit; wait a few minutes.
No drone-found notificationFirst check the module is running the 15 July or later firmware (earlier builds advertised the UUID where phone scan filters cannot see it, so detection never worked). Then confirm it is in NORMAL mode: it only advertises when being pinged, and drops to FALLBACK (BLE off) if the Pi stops. Also make sure the phone's Bluetooth is on before arming.
"Watch armed" toggles but nothing else happensThat is expected: arming does NOT turn the phone's Bluetooth on or off. It starts a background scan and shows the persistent "watch active" notification. Detection then produces a separate high-priority alert when a module is in range.
Notification never arrives while lockedSome Android skins aggressively kill background apps. Grant the app "unrestricted" battery usage in Android settings, then re-arm.
SOS button stays grey after joining Wi-FiGive it a few seconds — the app re-checks connectivity every 4 seconds and lights up on its own. If it stays grey, confirm the phone is really on RESCUE_A (open http://10.42.0.1/ in the browser). The one-flow trap that used to require the notification path is fixed in this build.
Rescue app kicks me to the login screen from HQ UplinkFixed in the 15 July build; make sure you re-downloaded the rescue APK.
Android warns the app is unsafeExpected for sideloaded test builds signed with a debug key. Choose install anyway.
