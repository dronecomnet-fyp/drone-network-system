Site, AI, Planning, Fleet

Verify the whole HQ-to-field mission layer: the live product site, the GCC spec fetch, the AI planner, rescuer tracking, and the fleet manager. Each stage is a gate. Drones stay props-off throughout.

01

The product site is live and data-backed

Open the site in any browser (online). It is a real React app backed by Supabase, not a static mockup.

  • Catalog shows three products: DroneComNet Module, Aux Module, AeroSync 5.
  • Open AeroSync 5: the 3D model rotates, and the spec table lists the CC3D flight controller, 2200 kV motors, AP/mesh ranges.
  • Unit lookup: enter DRN-S-0007 → it resolves to AeroSync 5 with the full spec JSON.

Pass when

The catalog, the 3D view, the unit lookup for DRN-S-0007, and a quote submission all work. If the page shows a plain README instead, GitHub Pages is still serving the branch: set Pages source to GitHub Actions and re-run the "Deploy product website" workflow.

02

The GCC fetches unit specs into a mission

In the GCC (online for this step), open the Mission tab. Create a mission (name it, e.g. "Flood 2026"). Add a comm module by unit ID, then a drone.

  • Add module DCM-A-0042. Tap its fetch specs (cloud) icon → snackbar "Fetched ... for DCM-A-0042"; the icon turns to a filled cloud.
  • Add an our-brand drone with unit ID DRN-S-0007, then fetch its specs → the row shows the AP range from the product site.
  • Add a volunteer drone ("DJI Mavic", pilot name) and attach module DCM-A-0042; its coverage now resolves from the module.
Behind the scenes
The GCC calls Supabase for units?unit_id=eq.DRN-S-0007&select=...,products(*) with the public anon key and caches the result into the mission file, so once fetched a unit resolves with no network.
Pass when

Specs appear on the drone/module rows and persist after you save and reload the mission file. Offline, a previously-fetched unit still resolves; a never-fetched one shows "specs unknown" (enter them manually, by design).

03

Draw the area, then AI-suggest a deployment

On the Map tab, turn on Planning, then Draw area and tap a polygon around your demo location; tap Done area. Back on the Mission tab, tap AI suggest (online; the OpenRouter key must be in Settings).

  • A short "Asking the AI..." dialog, then an AI plan N (draft) appears with a summary and any warnings.
  • It becomes the active deployment: the placements show on the map as role-colored pins with coverage circles.
  • Edit one (select in the panel, tap a new spot to move it), then tick approve on the Mission tab.
The validator is the guarantee
The AI only proposes JSON. The GCC validates it before rendering: placements outside the polygon, more than the available drones, a second system drone, or a node out of mesh range each raise a warning. Verified from here: a 3-drone request returned 3 placements, all inside the area, one system drone.
Pass when

An AI draft lands, you can edit and approve it, and it never commands a drone. Errors are honest: offline → "AI planning needs internet"; bad key → 401; free tier busy → 429. If a model is unavailable, pick another free model in Settings (the OpenRouter preset uses a current one).

04

A rescuer appears on the map

This uses the location heartbeat (rolled out per the node-update runbook). On the rescue app, log in and leave it open in the foreground; keep Settings → Share my location on.

  • Within about 90 s the rescuer shows on the GCC Live Ops Rescuers card with a fresh "updated" age, and as a teal person pin on the Map.
  • Toggle the app's Share my location off and on to show the control; background the app to show it pauses (battery).
Pass when

The rescuer's position appears on the GCC and updates. If not: confirm the nodes were updated (the personnel_locations table exists), the phone is on a RESCUE_x Wi-Fi with the fleet CA loaded, and location permission is granted. The GCC must be logged in (or have the break-glass key) to read locations.

05

Fleet: deploy in DEMO, watch the auto-return

On Live Ops, the fleet board shows "Deployed X / Y available" from your inventory. With no live MAVLink link, deploys are DEMO simulations, safe for any number of drones.

  • Deploy a drone to a placement (leave "command over MAVLink" off). It flies out (ENROUTE), holds (ON STATION), and when the modelled battery reaches the home-reserve it turns back (RETURNING) and lands, freeing its slot.
  • Deploy until the pool is empty → the Deploy button disables with "No more drones available".
  • On a flying drone tap simulate signal loss → it drops to FALLBACK (aux LoRa beacon at last position), then LOST after the timeout.
  • Volunteer rows read pilot advisory: the GCC shows the instruction rather than commanding flight.
Pass when

Drones fly, auto-return before the battery dies, the availability count gates Deploy, and signal loss shows FALLBACK then LOST. This is the "manage 10 drones and bring them home" story, exam-safe.

06

Fleet: real DRONE_S deploy, props off

Props off
  • All four props removed, frame secured, before the flight battery goes in.
  • This is command-pipeline verification, NOT flight. The drone is not flown.

Connect the GCC to DRONE_S on the Drone tab (DIRECT on RESCUE_S, or RELAY via a volunteer). With the heartbeat fresh (under 2 s), the fleet board's Deploy dialog enables command over MAVLink.

  • Deploy DRONE_S for real: the GCC runs the GUIDED → arm → takeoff → reposition sequence. On the bench the FC accepts and acks each step; the motor test spins a motor.
  • The battery watchdog issues return-to-launch below the per-cell threshold; force DISARM is always visible.
  • Only DRONE_S is real; every other drone in the fleet is DEMO (only one has a flight controller).
Pass when

The command sequence is observed on the bench (acks, motor spin, watchdog would RTL), props off, with force-DISARM reachable. Free flight stays out of scope until you clear it deliberately.

07

Live Ops is the single operational picture

Open Live Ops and confirm the whole operation reads at a glance, each figure with its own data age:

  • Stat tiles: victim messages NEW/CLAIMED, SOS + check-ins, field reports, rescuers tracked, mesh nodes, node battery/GPS, clock source.
  • The fleet board (stage 5/6) and the Rescuers card (stage 4).
  • The known-nodes table: connected node, peers, any DEGRADED node.
  • On the Map: victims (red/green), SOS (orange), rescuers (teal), the node (cyan), degraded nodes (red), and the deployed drones moving through their lifecycle. Planned placement vs actual drone GPS is the planned-vs-actual view.
Pass when

Every number carries an age and nothing pretends to be more live than the mesh allows. That honesty is the point: the operator always knows how fresh each piece of the picture is.

08

Troubleshooting

  • Site shows the README, not the app. Pages is serving the branch. Settings → Pages → Source → GitHub Actions, then re-run the "Deploy product website" workflow (Actions tab).
  • Fetch specs says "not configured". The GCC ships the public Supabase URL + anon key as defaults; if you cleared them, re-enter in Settings → Product site.
  • AI suggest disabled or errors. Paste the OpenRouter key in Settings → AI deployment advisor (endpoint and model are pre-filled). 401 = bad key; 429 = free tier busy, retry; "model unavailable" = choose another free model.
  • No rescuer on the map. Nodes must have the location update (personnel_locations table); the phone must be on a RESCUE_x Wi-Fi with the fleet CA; the GCC must be logged in to read locations.
  • Real deploy greyed out. The MAVLink link is not fresh: connect on the Drone tab and confirm a heartbeat under 2 s. DEMO always works without a link.
