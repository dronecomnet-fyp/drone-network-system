Node Update
Rescuer Location Sharing

Roll one small software update to each node so the ground control app can see where rescue teams are. New signed table, syncs fleet-wide like messages. No reflash, no data loss, about five minutes per node.

Nodes DRONE_A, DRONE_B, DRONE_S Adds personnel_locations table + endpoints Downtime a few seconds per node
What this does, and why it is safe

The rescue app already knows GPS. This update lets a logged-in rescuer send a periodic location heartbeat to the node, which signs it, stores the latest one per rescuer, and syncs it to the other nodes. The GCC then shows each rescuer's last known position on the map.

The new database table is created automatically the first time the updated API starts (CREATE TABLE IF NOT EXISTS). Nothing is dropped and no existing data is touched. Do the nodes one at a time; the fleet keeps working throughout.

Stages

  1. 1Update one node
  2. 2Gate — table + services
  3. 3Roll to the other nodes
  4. 4Gate — fleet-wide sync
  5. 5Gate — shows on the GCC
  6. 6Troubleshooting
01

Update one node (start with DRONE_A)

Join that node's Wi-Fi (or reach it over Ethernet), SSH in, pull the new code, and restart just the API and sync services. The captive portal and Wi-Fi stay up.

SSH · DRONE_Adrone@10.42.0.1
$ ssh drone@10.42.0.1
$ cd ~/rescue-mesh
$ git pull
$ sudo systemctl restart rescue-mesh-api rescue-mesh-sync

No dependency changes ship with this update, so there is nothing to pip install. If git pull reports local changes, you are on a modified node: stash or review before pulling, do not force.

02

Gate: the table exists and services are healthy

Confirm the API came back up and created the new table. /health lists a count per replicated table.

SSH · DRONE_A
$ curl -sk https://10.42.0.1:8443/health | grep -o personnel_locations
$ systemctl is-active rescue-mesh-api rescue-mesh-sync
$ journalctl -u rescue-mesh-sync -n 20 --no-pager
Expect
personnel_locations
active
active
The word personnel_locations appears in /health (the table is registered and counted), both services are active, and the sync log shows peers syncing with no tracebacks.
Stop here

If /health does not mention personnel_locations, the old code is still running: re-check git pull landed and restart the API again. Do not move to the next node until this one is clean.

03

Roll the same update to DRONE_B and DRONE_S

Repeat stages 1 and 2 on each remaining node. Mixed versions are fine during the roll: an un-updated node simply does not have the new table yet and ignores the new sync path until you update it. Order does not matter.

SSH · DRONE_B then DRONE_S
$ ssh drone@10.42.0.1        # on each node's own AP
$ cd ~/rescue-mesh && git pull
$ sudo systemctl restart rescue-mesh-api rescue-mesh-sync
$ curl -sk https://10.42.0.1:8443/health | grep -o personnel_locations
DRONE_S is no different here. It runs the same node software, so the update and the gate are identical; its MAVLink gateway is untouched by this change.
04

Gate: a location filed on one node reaches another

This proves the feature end to end: sign, store, and sync. Post a location as a logged-in rescuer on DRONE_A, then read it back from DRONE_B after one sync cycle.

On DRONE_A: post as a rescuer
$ TOKEN=$(curl -sk https://10.42.0.1:8443/auth/login \
    -H 'Content-Type: application/json' \
    -d '{"personnel_id":"R-001","pin":"YOUR_PIN"}' | python3 -c 'import sys,json;print(json.load(sys.stdin)["token"])')
$ curl -sk https://10.42.0.1:8443/personnel-location \
    -H "X-Session-Token: $TOKEN" -H 'Content-Type: application/json' \
    -d '{"lat":6.9271,"lon":79.8612,"battery_pct":80}'

Wait one sync interval (about 30 s), then on DRONE_B read the fleet-wide list (any role key or session token works):

On DRONE_B: read it back
$ curl -sk https://10.42.0.1:8443/personnel-locations \
    -H "X-API-Key: $HQ_KEY"
Expect on DRONE_B
[{"personnel_id":"R-001","lat":6.9271,"lon":79.8612,
  "battery_pct":80,"node_id":"DRONE_A", ...}]
R-001's position, filed on A, is now readable on B with node_id: DRONE_A: it was signed, synced, and verified across the mesh. A tampered copy would be rejected at sync and never appear.
Stop here

If it never appears on B, check that both nodes passed stage 2, that they see each other (/health peers), and that the sync log shows no rejects. A wrong PIN gives 401 at login, not a sync problem.

05

Gate: rescuers show up on the ground control app

Update the rescue app to the build with location sharing, log in, and leave it open in the foreground. In the GCC, open the Live Ops tab (Rescuers card) and the Map (teal person pins).

  • The rescue app Settings has a Share my location switch (on by default) showing when it last sent.
  • Within about 90 seconds the rescuer appears in Live Ops with a fresh "updated" age, and as a teal pin on the map.
  • Battery-friendly by design: it only sends while logged in and in the foreground. Background and logged-out send nothing; the operator sees the last known position with its age.
Done
Rescuers now appear on the operations picture alongside victims, SOS check-ins, and drones. Each carries its own data age, so the operator is never misled about how fresh a position is.
06

Troubleshooting

  • App shows "Cannot reach the drone" when sharing. The heartbeat uses the same authenticated API as everything else: confirm the phone is on a RESCUE_x Wi-Fi and the fleet CA is loaded. It shares the message poller's back-off, so it will not spam on failure.
  • Location switch on but nothing sent. Grant the app location permission and enable device location. The app uses medium accuracy and falls back to the last known fix; indoors with no fix it simply waits.
  • Rescuer never appears on the GCC. The GCC reads /personnel-locations only once credentials are set: log in (or set the break-glass key) in GCC Settings. Then confirm the node passed stage 4.
  • Old position lingers. That is the honest last-known behaviour: the age next to it shows how stale it is. It refreshes on the next heartbeat.
