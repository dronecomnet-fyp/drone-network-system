System Drone Node DRONE_S
Mesh Peer + MAVLink Gateway

Bring up the third Pi as a full mesh node on the drone, wired to the flight controller. It syncs with A and B, serves the drone's own GPS into the network, and lets the GCC spin the motors, direct or relayed through a volunteer drone. Props off.

Node DRONE_S (RESCUE_S, 10.99.0.3) Needs Pi + 2nd AR9271 + serial wire to the FC No ESP32, no aux module
Two rules carried over from earlier runbooks

Copy DRONE_A's deploy/secrets/ to DRONE_S; do NOT run make_fleet_ca.sh. Same as DRONE_B: a different key set means the nodes silently reject each other and never sync.

Props OFF for everything here. The last stages spin motors from the GCC. Remove all four props and secure the frame before you connect the flight battery.

00

Hardware: adapter in, ESP32 out, FC wired to the Pi

  1. Plug the second AR9271 USB adapter into the drone's Pi. This is the 2.4 GHz mesh radio; the Pi's onboard WiFi becomes RESCUE_S.
  2. Remove the ESP32. It was only the FC's WiFi-serial bridge; the Pi replaces it. Nothing else on the drone changes and no firmware is reflashed.
  3. Wire the Pi to the CC3D telemetry UART. Both are 3.3 V, so connect directly, no level shifter:
    • CC3D telem TX → Pi pin 10 (GPIO15, RX)
    • CC3D telem RX → Pi pin 8 (GPIO14, TX)
    • CC3D GND → any Pi GND pin
    This gives /dev/serial0. A USB-TTL adapter (/dev/ttyUSB0) is the alternative; set FC_SERIAL to match in the config.
  4. Props OFF. Note the FC telemetry baud (CC3D telem is commonly 57600); you will set FC_BAUD to it.
If the ESP32 was using the FC's only telemetry UART, the Pi now uses that same port (you moved the wires from ESP32 to Pi). The FC is already emitting MAVLink there, so no FC change is needed.
01

Flash and boot drone-s

Same as the other nodes, hostname drone-s: Raspberry Pi OS Lite (64-bit), enable SSH, user drone, timezone Asia/Colombo, no Wi-Fi. Connect Ethernet (setup takes over the onboard Wi-Fi), power on, and SSH in:

PowerShell
PS> ssh drone@drone-s.local

Keep DRONE_A powered on the same network for the secrets copy in the next stage.

02

Clone the repo, copy A's secrets

drone-s shellDRONE_S
$ git clone https://github.com/dronecomnet-fyp/drone-network-system.git ~/rescue-mesh
$ scp -r drone@drone-a.local:~/rescue-mesh/deploy/secrets ~/rescue-mesh/deploy/
$ ls ~/rescue-mesh/deploy/secrets   # fleet_ca.crt fleet_ca.key fleet_secrets.env
Do NOT run make_fleet_ca.sh here. DRONE_S must share A/B's keys to join the mesh.
03

Gate — both radios and the FC serial

drone-s shellDRONE_S
$ lsusb | grep -i atheros          # the 2nd AR9271 is present
$ iw list | grep -A8 "Supported interface modes"   # must list IBSS
$ ls -l /dev/serial0               # the FC UART (GPIO path)
Stop here

Confirm the AR9271 shows in lsusb, IBSS is in the interface modes, and the FC serial device exists (/dev/serial0 for GPIO, or /dev/ttyUSB0 for USB-TTL). If the serial console still owns /dev/serial0, the setup script frees it in the next stage.

04

Run setup as node s

If your FC serial or baud differ from the defaults (/dev/serial0, 57600), edit deploy/nodes/drone_s.conf first. Then:

drone-s shellDRONE_S
$ cd ~/rescue-mesh/deploy
$ chmod +x setup_node.sh make_fleet_ca.sh files/dtn-net-up.sh
$ sudo ./setup_node.sh s

Beyond the normal node setup, the s role also installs the MAVLink gateway (pymavlink, the rescue-mesh-mavgw service), frees the UART from the login console, and adds the drone user to dialout. It ends with === Setup complete for DRONE_S ===.

05

Reboot

drone-s shellDRONE_S
$ sudo reboot

The UART and interface-pinning changes need the restart. Wait a minute, then reconnect and connect the drone's flight battery so the FC powers up and starts sending MAVLink.

06

Gate — DRONE_S alive, with the drone's GPS in health

drone-s shellDRONE_S
$ systemctl is-active rescue-mesh-api dtn-net rescue-mesh-sync rescue-mesh-mavgw
$ curl -sk https://10.42.0.1:8443/health | python3 -m json.tool
Expected /health (drone outdoors / near a window)
"node_id": "DRONE_S",
"gps": { "lat": 6.92..., "lon": 79.86..., "fix": 1, "sats": 8 },
"battery": { "a_v": 11.4, ... },   (flight battery, from the FC)
"clock_source": "gps"

The GPS and battery here come from the flight controller through the gateway, not an aux module. If rescue-mesh-mavgw is not active, check the FC wire and journalctl -u rescue-mesh-mavgw -n 20 (look for MAVGW_FC_CONNECT).

Stop here

Confirm all four services active, node_id is DRONE_S, and the GPS/battery are populated from the FC once it has a fix. That proves the node AND the gateway telemetry tap.

07

Gate — mesh sync with A and B

Power A (and B) nearby. From DRONE_S, reach across the mesh and confirm sync, exactly like the two-node test:

drone-s shellDRONE_S
$ ping -c 3 10.99.0.1          # DRONE_A over the mesh
$ curl -sk https://10.42.0.1:8443/health | python3 -m json.tool | grep -A4 peers
Stop here

Confirm the ping works and DRONE_A/B appear as peers. Send a victim message on RESCUE_A and confirm it reaches DRONE_S within ~40 s (curl /messages with the HQ key). DRONE_S is now a first-class mesh node.

08

Gate — control the drone directly from the GCC

Props off, secured
  • All four props removed, frame clamped, area clear, flight battery connected, kill switch (red DISARM) understood.
  1. On the laptop, join RESCUE_S (the drone's own 5 GHz AP).
  2. In the GCC Settings, set the MAVLink target with the DIRECT (RESCUE_S) chip (udp:10.42.0.1:14550).
  3. Drone tab → Connect. Confirm the heartbeat card goes green and telemetry matches Stage 6.
  4. Tap Motor 1, confirm the dialog: motor 1 spins briefly, props off.
Stop here — motors from the GCC, over the mesh node

A motor spun on a GCC command, through the drone's own Pi gateway. If anything is off, hit DISARM or pull the battery. Follow the full-detail drone-control runbook for the command palette and arming notes.

09

Gate — control relayed through a volunteer drone

This is the headline: the GCC controls the system drone without joining its network, routed live across the mesh through a volunteer drone.

  1. Keep DRONE_S and a volunteer (DRONE_A) powered and in mesh range.
  2. On the laptop, join RESCUE_A (the volunteer's AP), NOT RESCUE_S.
  3. In GCC Settings, set the target with the RELAY (via mesh) chip (udp:10.99.0.3:14550).
  4. Drone tab → Connect. The heartbeat should come up, routed laptop → volunteer Pi → mesh → DRONE_S gateway → FC.
  5. Props off: run a motor test again to confirm commands make it across.
Stop here — this is the whole file-08 story

Commanding the system drone through a volunteer drone's module, live (not DTN), is the capstone. Note the small extra latency of the two hops; that is expected and fine for telemetry and guided commands. If the heartbeat greys out (>2 s), the GCC disables commands automatically, which is the safety gate working.
