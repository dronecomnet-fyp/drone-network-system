Second Node DRONE_B
and the Two-Node Mesh

Build a second Pi as DRONE_B, join it to DRONE_A on the 2.4 GHz backbone, and prove the headline feature: a message sent at one drone reaches a rescuer at the other, with no internet and no central server.

New node DRONE_B (RESCUE_B, 10.99.0.2) Needs Pi + AR9271 + aux module Drive from Windows (SSH over Ethernet)
victim at DRONE_A→ 2.4 GHz mesh sync→ rescuer at DRONE_B claims it→ claim syncs back to A
Read this first — the one mistake that breaks everything

Every node in the fleet shares one set of trust files: the same master secret (which derives the signing keys) and the same certificate authority. DRONE_B must copy DRONE_A's deploy/secrets/ folder.

Do NOT run make_fleet_ca.sh on DRONE_B. If you do, B gets its own keys, and the two nodes will silently reject each other's beacons and records as forged. Everything will look configured, yet nothing will ever sync, with no obvious error. Stage 3 does the copy; that is the crux of this whole runbook.

00

Prepare aux module B (flash + provision)

The second soldered XIAO becomes DRONE_B's aux module. Do this on the Windows laptop with the same steps as the first module (the aux runbook), with two specifics:

  1. Flash the CURRENT firmware (git pull first). Firmware from before 15 July advertised Bluetooth where phone scan filters cannot see it, so the emergency app would never detect this module. If module A has not been re-flashed since then either, re-flash it too while you are set up.
  2. Provision it as DRONE_B, not DRONE_A. In aux_sim.py:
PowerShell · firmware\aux1 then tools\
PS> git pull
PS> pio run -t upload
PS> cd ..\..\tools
PS> python aux_sim.py --port COM5
n DRONE_B
Expected
<- {"type":"set_node_id_ack","node_id":"DRONE_B"}

The identity is stored in the module's flash and survives power cycles and reflashes. It is what the module beacons as in fallback mode (FB|DRONE_B|...) and advertises over Bluetooth (B|RESCUE_B), so getting the letter right matters. Quick sanity check with nRF Connect on a phone: the service data should now read B|RESCUE_B.

Already flashed it earlier "like module A"? Then just confirm two things: it was flashed with the post-15-July firmware (re-flash if unsure — it is one command), and its ack says DRONE_B, not DRONE_A. Two modules claiming the same identity would poison the fallback beacons.
01

Flash the DRONE_B SD card

Exactly like DRONE_A, with one change: the hostname. In Raspberry Pi Imager, Raspberry Pi OS Lite (64-bit), Edit Settings:

  • Hostname: drone-b
  • Enable SSH, username drone + a password, timezone Asia/Colombo
  • Do NOT configure Wi-Fi

Put the card in the second Pi, and plug in its AR9271 USB adapter and the aux module you just provisioned as DRONE_B in Stage 0 (near a window, for GPS).

02

Boot and SSH in over Ethernet

Connect DRONE_B by Ethernet (same reason as before: setup takes over its Wi-Fi). Power on, wait a minute, then from Windows:

PowerShell
PS> ssh drone@drone-b.local

Leave DRONE_A powered and on the same network too — Stage 3 copies files from it.

03

Get the code, then copy A's secrets

On DRONE_B, clone the repo:

drone-b shellDRONE_B
$ git clone https://github.com/dronecomnet-fyp/drone-network-system.git ~/rescue-mesh

Now copy the trust files from DRONE_A (this is the step that makes the mesh work):

drone-b shellDRONE_B
$ scp -r drone@drone-a.local:~/rescue-mesh/deploy/secrets ~/rescue-mesh/deploy/
Verify B now has A's secrets, and did NOT generate its own
$ ls ~/rescue-mesh/deploy/secrets
fleet_ca.crt  fleet_ca.key  fleet_secrets.env
$ # the master secret here MUST equal DRONE_A's:
$ grep NODE_MASTER_SECRET ~/rescue-mesh/deploy/secrets/fleet_secrets.env

If that folder is empty or missing, the copy failed — fix it before going on. If you accidentally already ran make_fleet_ca.sh on B, delete the folder and re-copy from A: rm -rf ~/rescue-mesh/deploy/secrets then repeat the scp.

04

Gate — network stack and AR9271

Same checks as DRONE_A. On DRONE_B:

drone-b shellDRONE_B
$ systemctl is-active NetworkManager   # active
$ iw list | grep -A8 "Supported interface modes"   # must list IBSS
Stop here

Confirm IBSS appears, same as on A. No point running setup until the second adapter can do ad-hoc mode.

05

Run setup as node b

The b argument selects DRONE_B's config: SSID RESCUE_B, mesh address 10.99.0.2. It reads the secrets you copied and issues B its own node certificate from the shared fleet CA.

drone-b shellDRONE_B
$ cd ~/rescue-mesh/deploy
$ chmod +x setup_node.sh make_fleet_ca.sh files/dtn-net-up.sh
$ sudo ./setup_node.sh b

It ends with === Setup complete for DRONE_B ===. Note there is no make_fleet_ca.sh line here: you already have the fleet's secrets from A.

06

Reboot

drone-b shellDRONE_B
$ sudo reboot

Wait a minute, reconnect: ssh drone@drone-b.local.

07

Gate — DRONE_B is alive on its own

drone-b shellDRONE_B
$ systemctl is-active rescue-mesh-api rescue-portal dtn-net rescue-mesh-sync
$ ls /dev/ttyACM*                          # the aux module enumerated
$ curl -sk https://10.42.0.1:8443/health | python3 -m json.tool
Expected /health highlights
"node_id": "DRONE_B",
"aux": "present",
"gps": {... "sats" climbing, "fix": 1 near a window ...},
"clock_source": "gps"   (after a fix; "relative" before)

If aux says absent, check the module is on /dev/ttyACM0 and restart the bridge: sudo systemctl restart rescue-mesh-auxbridge. journalctl -u rescue-mesh-auxbridge -n 10 should show AUX_CONNECT and, after a fix, CLOCK_SYNC.

Stop here

Confirm the services are active, /health reports "node_id":"DRONE_B", and aux is present with GPS data flowing. B is now a complete node in its own right, aux included; next we link the two.

08

Gate — the 2.4 GHz mesh link

Put both Pis within a couple of metres of each other and leave both powered. The IBSS cell forms on its own (each node's dtn-net service). From DRONE_A, reach across the mesh to DRONE_B:

drone-a shellDRONE_A
$ iw dev wlan1 info          # type IBSS, ssid RESCUE_DTN
$ ping -c 3 10.99.0.2        # DRONE_B's mesh address
Expected
64 bytes from 10.99.0.2: icmp_seq=1 ttl=64 time=3.1 ms
64 bytes from 10.99.0.2: icmp_seq=2 ttl=64 time=2.8 ms

Ping working means the two AR9271 adapters have joined the same ad-hoc cell and can carry traffic. This is the physical layer the sync rides on.

Stop here

Confirm the ping succeeds both ways (also try ping 10.99.0.1 from DRONE_B). If it fails, both radios are not on the cell yet — see troubleshooting; do not expect sync until the ping works.

09

Gate — prove they are syncing

The sync daemons announce themselves over the mesh every 10 s and pull changes every 30 s. First, confirm each sees the other as a peer:

drone-a shellDRONE_A
$ curl -sk https://10.42.0.1:8443/health | python3 -m json.tool | grep -A6 peers
Expected: DRONE_B listed as a peer
"peers": [
    { "node_id": "DRONE_B", "ip": "10.99.0.2", "last_seen": "..." }
]

Now the real test. Create a record on B and watch it arrive on A. Use B's break-glass key to file a quick field report:

drone-b shellDRONE_B
$ HQ=$(grep HQ_API_KEY ~/rescue-mesh/deploy/secrets/fleet_secrets.env | cut -d= -f2)
$ curl -sk -H "X-API-Key: $HQ" -H "Content-Type: application/json" \
       -d '{"content":"mesh sync test from B"}' https://10.42.0.1:8443/gs-uplink

Wait about 40 seconds, then on DRONE_A:

drone-a shellDRONE_A
$ HQ=$(grep HQ_API_KEY ~/rescue-mesh/deploy/secrets/fleet_secrets.env | cut -d= -f2)
$ curl -sk -H "X-API-Key: $HQ" https://10.42.0.1:8443/gs-messages | grep "from B"
Stop here — this is the thesis in one command

If "mesh sync test from B" appears on DRONE_A, the delay-tolerant sync works: a record created on one node replicated itself across the ad-hoc radio to the other, verified its signature, and stored it. Peek at tail ~/rescue-mesh/backend/audit.log on either node to see the SYNC_OK lines. If nothing arrives, that is almost always the secrets mistake from the top — check troubleshooting.

10

The full demo loop (optional, very satisfying)

With sync proven, run the story end to end across the two drones:

  1. A phone joins RESCUE_A and sends a victim message through the captive portal.
  2. A rescuer's phone joins RESCUE_B and logs into the rescue app with a PIN (the personnel record was issued on A and synced to B — the login works with no re-issuing).
  3. The victim message, submitted at A, appears in the rescuer's list at B. They claim it.
  4. Back at A, the claim shows the rescuer's ID: the claim synced B → A.

That is a victim helped at one drone by a rescuer standing at another, over a self-forming radio mesh with no internet anywhere. It is the strongest single thing to show an examiner.

You can also point the GCC at each node in turn (join RESCUE_A, then RESCUE_B) and watch the same messages, personnel, and announcements on both — proof the whole dataset converges.

Troubleshooting

SymptomTry this
Ping 10.99.0.2 failsOn each node iw dev wlan1 info must show type IBSS, ssid RESCUE_DTN, the same fixed BSSID and frequency. If one is not joined, sudo systemctl restart dtn-net and re-check. Keep the Pis close for the first test.
Ping works, but no peer appears and nothing syncsAlmost certainly the secrets mistake: B has different keys from A. On A, tail -f ~/rescue-mesh/backend/audit.log and look for BEACON_REJECT or SYNC_REJECT. If present, re-copy deploy/secrets from A to B (Stage 3) and re-run sudo ./setup_node.sh b, then reboot.
Peer appears but records rejected (SYNC_REJECT)Same root cause as above (mismatched signing key), or a mismatched fleet CA so TLS between nodes fails. Re-copying the whole secrets folder fixes both.
DRONE_B /health says node_id DRONE_AYou ran setup with the wrong letter. Re-run sudo ./setup_node.sh b and reboot.
Module B identifies as DRONE_A (nRF Connect shows A|RESCUE_A, or fallback beacons say FB|DRONE_A)The module was provisioned with the wrong identity. Connect it to the laptop, run aux_sim.py, send n DRONE_B, confirm the ack (Stage 0). Note: while plugged into Pi B, the bridge also pushes B's identity on connect, which can mask this until fallback mode.
Sync is slow to appearNormal: beacons every 10 s, pulls every 30 s, so allow up to ~40 s. It is delay-tolerant by design, not instant.
