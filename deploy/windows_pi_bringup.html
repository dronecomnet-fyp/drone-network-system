00

What you're building

One complete node called DRONE_A, matching the aux module you provisioned as DRONE_A. On this single Pi:

  • Onboard WiFi (wlan0) becomes the 5 GHz user access point RESCUE_A at 10.42.0.1 — what victims and rescuers connect to.
  • USB AR9271 (wlan1) becomes the always-on 2.4 GHz mesh radio (the DTN backbone). With one Pi it has no peer yet, which is fine — it just waits.
  • The XIAO aux module plugs into a USB port and feeds GPS + battery telemetry to the node over serial (last stage).
With only one node, you can fully verify a working node and the aux integration. Node-to-node mesh sync is a later step that needs a second Pi.
01

Flash the SD card

On the Windows laptop, install Raspberry Pi Imager, insert the SD card, and open it.

  1. Choose Device: Raspberry Pi 4.
  2. Choose OS: Raspberry Pi OS (other) → Raspberry Pi OS Lite (64-bit). Lite = no desktop, which is what we want.
  3. Choose Storage: your SD card.
  4. Click Next, then Edit Settings, and set:
    • Hostname: drone-a
    • Enable SSH (Services tab) — password authentication is fine to start.
    • Username drone, and a password you'll remember.
    • Locale: timezone Asia/Colombo.
    • Do NOT configure wireless LAN — leave WiFi blank. The setup script owns the radios.
  5. Save, then Write. When it finishes, put the card in the Pi.
02

Boot and SSH in

  1. Plug the AR9271 USB WiFi adapter into the Pi.
  2. Connect the Pi to your router with an Ethernet cable.
  3. Power on. Give it about a minute to boot the first time.

From Windows PowerShell:

PowerShellWindows
PS> ssh drone@drone-a.local
Expected
The authenticity of host 'drone-a.local ...' can't be established.
Are you sure you want to continue connecting (yes/no)? yes
drone@drone-a.local's password: (type your password)
drone@drone-a:~ $

If drone-a.local doesn't resolve, find the Pi's IP address in your router's device list and use ssh drone@THE.IP.ADDRESS instead.

04

Gate — network stack and AR9271

The setup script refuses to run if the network stack is wrong, so check it first. On the Pi:

Pi shellPi · SSH
$ systemctl is-active NetworkManager   # expect: active
$ systemctl is-active dhcpcd           # expect: inactive (or failed)
$ lsusb                                # expect an Atheros / AR9271 line
$ dmesg | grep -i ath9k                # expect ath9k_htc firmware loaded
$ iw list | grep -A8 "Supported interface modes"
The one that matters most
Supported interface modes:
     * managed
     * AP
     * IBSS          ← this line must be present
     * monitor

IBSS is the ad-hoc mode the 2.4 GHz backbone uses. If dmesg shows the ath9k firmware missing, install it: sudo apt install firmware-ath9k-htc, then re-check.

Stop here

Report the results, especially whether IBSS appears in the interface modes and whether the Atheros adapter shows in lsusb. These two are the foundation of the dual-radio design; if either is missing we sort it out before running setup.

05

Generate the fleet trust files (once)

This creates the certificate authority and shared secrets the whole fleet uses. Run it once; every node reuses the same deploy/secrets/ folder. On the Pi:

Pi shellPi · SSH
$ cd ~/rescue-mesh/deploy
$ chmod +x setup_node.sh make_fleet_ca.sh files/dtn-net-up.sh
$ ./make_fleet_ca.sh
Expected tail
[1/3] Generating fleet CA key + certificate
[2/3] Generating fleet secrets
[3/3] Done. Copy deploy/secrets/ to each Pi ...

This produces deploy/secrets/ with fleet_ca.crt (the apps will need this later), fleet_ca.key, and fleet_secrets.env. They're git-ignored on purpose — never commit them.

06

Run the node setup

The a argument means DRONE_A (the config for SSID RESCUE_A, DTN address 10.99.0.1, aux on /dev/ttyACM0). The interface MAC addresses are detected automatically, so there's nothing to edit.

Pi shellPi · SSH
$ sudo ./setup_node.sh a
What you'll see scroll by
[Step 2] Verifying network stack assumptions
[Step 3] Pinning interface names (wlan0=onboard, wlan1=AR9271)
[Step 6] Installing base packages
[Step 6] Python venv + pinned requirements
[Step 7] Writing /etc/rescue-mesh/node.env
[Step 7] Issuing node TLS certificate from the fleet CA
[Step 4] Configuring 5 GHz user AP (RESCUE_A, channel 36)
[Step 5] Configuring IBSS DTN backbone on wlan1
[Step 8] ... systemd units ... firewall ... disabling Bluetooth
=== Setup complete for DRONE_A ===
REBOOT NOW ...

The package install and pip step need internet — that's why the Pi is on Ethernet. This takes a few minutes.

If the pip step fails: the Python package versions were pinned on a newer Python than the Pi's 3.11. If you see a pip resolver error here, stop and tell me the error — it's a one-line fix to the version pins, and worth doing properly rather than hacking around.
07

Reboot

The interface-name pinning and the Bluetooth-off change only take effect after a restart.

Pi shellPi · SSH
$ sudo reboot

Your SSH session drops. Wait about a minute, then reconnect: ssh drone@drone-a.local.

08

Gate — the node is alive

Back on the Pi after reboot:

Pi shellPi · SSH
$ ip link                              # wlan0 AND wlan1 both present
$ systemctl is-active rescue-mesh-api rescue-portal dtn-net rescue-mesh-sync
$ iw dev wlan1 info                    # type IBSS, ssid RESCUE_DTN
$ curl -sk https://10.42.0.1:8443/health
Expected /health (aux not plugged in yet)
{"status":"ok","node_id":"DRONE_A","aux":"absent",
 "gps":{"lat":null,"lon":null,"fix":0,"sats":0},
 "clock_source":"relative", ...}

aux":"absent" is correct here — the module isn't connected yet. Then grab your phone: the RESCUE_A WiFi network should be visible; joining it (it's open) should pop the captive portal with the victim message form.

Stop here

Report: are all four services active, does iw dev wlan1 info say type IBSS, does /health return JSON, and does the RESCUE_A network appear on your phone with the portal? If wlan1 won't become IBSS, tell me the exact error — there's a documented fallback and I'll walk you through it.

09

Gate — plug in the aux module

Now the two devices meet. Plug the XIAO (flashed and provisioned as DRONE_A) into a Pi USB port with the USB-A→USB-C cable. Put it near a window for GPS.

Pi shellPi · SSH
$ ls /dev/ttyACM*                      # expect /dev/ttyACM0
$ sudo systemctl restart rescue-mesh-auxbridge
$ curl -sk https://10.42.0.1:8443/health | python3 -m json.tool
Expected after a GPS fix (give it a few minutes)
"aux": "present",
"gps": {"lat": 6.9271, "lon": 79.8612, "fix": 1, "sats": 7},
"battery": {"a_v": 7.42, ...},
"clock_source": "gps"

Watch the bridge's own log for the moment it sets the clock from GPS:

Pi shellPi · SSH
$ journalctl -u rescue-mesh-auxbridge -n 20 --no-pager
# expect an AUX_CONNECT line, then after a fix a CLOCK_SYNC line
Stop here — this is the finish line for one module

Report the /health output once the module is connected: aux should read present, sats should climb, and after a fix clock_source flips to gps. That's a complete, self-locating node. Paste it back and we'll mark DRONE_A done.

10

Troubleshooting

SymptomTry this
SSH dies partway through setupYou were connected over WiFi. Use Ethernet and re-run sudo ./setup_node.sh a (it's safe to re-run).
setup_node.sh aborts: "NetworkManager is not active"Bookworm should have it by default. sudo systemctl enable --now NetworkManager and sudo systemctl disable --now dhcpcd, then re-run.
"missing fleet trust material"You skipped Stage 5. Run ./make_fleet_ca.sh from ~/rescue-mesh/deploy first.
pip / package install error during setupLikely the version pins vs the Pi's Python 3.11. Send me the exact error; it's a small fix to backend/requirements.txt.
wlan1 won't show type IBSSConfirm ath9k firmware loaded (dmesg | grep ath9k); if it flaps, tell me — there's a documented AP/station fallback for this exact case.
RESCUE_A WiFi not visible on the phoneOnboard 5 GHz AP needs the country code set (the script sets LK). Check nmcli con show USER_AP is active; some phones only see 2.4 GHz and won't list a 5 GHz AP (expected).
aux stays "absent" after plugging inls /dev/ttyACM* to confirm it enumerated; journalctl -u rescue-mesh-auxbridge -f to watch it connect; restart the service.
