00

Get the firmware files onto the Windows laptop

You only strictly need two files from firmware/aux/: platformio.ini and src/main.cpp. Copy the whole firmware/aux folder across — it's a few KB.

Any transfer method works: a USB drive, a shared cloud folder, or emailing the two files to yourself. Keep the same folder structure (firmware/aux/src/main.cpp, not flattened) — PlatformIO expects src/ as a subfolder next to platformio.ini.

Where to put it: anywhere without spaces or unusual characters in the path is safest — e.g. C:\rescue-mesh\firmware\aux.
01

Install Python

PlatformIO and the serial test tool both run on Python.

  1. Download Python 3.11 or later from python.org/downloads/windows.
  2. Run the installer. On the first screen, tick "Add python.exe to PATH" before clicking Install. This is the step people miss.

Verify in PowerShell:

PowerShell
PS> python --version
# expect: Python 3.11.x (or newer)

If PowerShell says it doesn't recognize python, close and reopen PowerShell first (PATH changes need a fresh window); if it still fails, re-run the installer and check the PATH box.

02

Install PlatformIO Core and pyserial

PlatformIO builds and flashes the firmware; pyserial runs the test tool that talks to the board afterward.

PowerShell
PS> pip install platformio pyserial

Verify:

PowerShell
PS> pio --version
# expect: PlatformIO Core, version 6.x

First run of pio after install downloads some tooling in the background — that's normal and only happens once.

03

Connect the board and find its port

Plug the XIAO into the laptop with the USB-A→USB-C cable. The XIAO ESP32-C3 uses native USB, so Windows 10/11 should enumerate it with no separate driver install.

PowerShell · from firmware\aux
PS> pio device list
Look for
COM5
------
Hardware ID: USB VID:PID=303A:1001 ...
Description: USB Serial Device (COM5)

The COM number will differ on your machine (COM3, COM7, whatever's free). Write it down — you'll use it if auto-detect ever fails below.

Nothing listed? See Troubleshooting before going further — don't proceed to flashing until a COM port shows up.

04

Build and flash

From inside the firmware\aux folder:

PowerShell
PS> cd C:\rescue-mesh\firmware\aux
PS> pio run
# first build downloads the ESP32 toolchain: several minutes, one time only
PS> pio run -t upload
Expected tail of the upload
Writing at 0x00010000... (100 %)
Wrote 720000 bytes...
Hard resetting via RTS pin...
================ [SUCCESS] Took NN.NN seconds ================

If pio run -t upload can't find the port or times out, specify it explicitly using the COM number from Stage 3:

PowerShell
PS> pio run -t upload --upload-port COM5

Still failing? Jump to Troubleshooting → boot mode — some first-time boards need the BOOT button held during plug-in.

05

Gate — confirm it booted before continuing

Open a live serial view:

PowerShell
PS> pio device monitor
Expected first line
{"type":"boot","node_id":"UNSET","lora":true,"ina3221":true}

lora and ina3221 report whether SPI/I2C found each chip at boot. Both should read true on your soldered module. Press Ctrl+C to leave the monitor when done.

Stop here

Report back exactly what the boot line says — especially if lora or ina3221 reads false, or nothing prints at all. Don't move to Stage 6 until this line looks right; a wrong value here points at a wiring or solder-joint issue that's easiest to fix now.

06

Provision this board's identity

One firmware image serves every module; each board is told which node it is over serial, once, and it remembers across reflashes. This board becomes DRONE_A.

Close pio device monitor first (only one program can hold the port). Then, from the repo's tools\ folder:

PowerShell · from tools\
PS> python aux_sim.py --port COM5

The tool starts pinging the board automatically and prints everything it says. At the prompt, type:

aux_sim.py
n DRONE_A
07

Gate — confirm the identity took

Expected
<- {"type":"set_node_id_ack","node_id":"DRONE_A"}
Stop here

Confirm the ack shows DRONE_A exactly. Leave aux_sim.py running — the next two stages read its output directly.

08

Verify GPS

Move the module near a window or outdoors — the GPS antenna needs sky. Watch aux_sim.py: a gps line arrives every 5 seconds.

Before a fix
{"type":"gps","fix":0,"lat":null,"lon":null,"sats":0}
After a fix (can take a few minutes cold)
{"type":"gps","fix":1,"lat":6.9271,"lon":79.8612,"sats":7,"hdop":1.1}

sats climbing from 0 and fix flipping to 1 is the pass condition. lat/lon should roughly match where you're standing.

09

Verify battery telemetry (INA3221)

The same battery line arrives every 5 seconds. Channel 1 is Battery A, channel 2 is Battery B — whichever channel currently has a supply wired to it should show a plausible reading; the other reads null if unpowered.

Expected shape
{"type":"battery","bat_a_v":7.42,"bat_a_ma":180,"bat_b_v":null,"bat_b_ma":null}

A voltage in a sane range for whatever you've wired to that channel is the pass condition. All four fields null on every line, even with a supply connected, points at an I2C wiring issue — check that against Troubleshooting.

10

Verify BLE advertising

Install nRF Connect (Nordic Semiconductor) on a phone from its app store, open it, and start a scan.

Expected in the scan list
Name: RESCUE-A
Service UUID: 2b57461c-1c04-49c4-944a-13643c1618da
Service Data: 41 7c 52 45 53 43 55 45 5f 41   ("A|RESCUE_A")

Advertising interval should sit around 500–1000 ms. If it isn't visible, confirm you're still in NORMAL mode — aux_sim.py must still be pinging (advertising stops automatically in FALLBACK mode).

11

LoRa — what's provable with one module

The full LoRa test needs a second flashed module to receive: one board transmits, the other prints lora_rx. With a single module right now, that receive-side confirmation isn't possible yet.

What you can confirm today: the boot line's "lora":true from Stage 5 already means LoRa.begin() succeeded — the SPI wiring to the RFM95 is correct and the radio answered. You can also send a transmit command and confirm it doesn't error:

aux_sim.py
l hello-bench

Nothing prints back for a lone transmit — no news is good news here. The real pass/fail (module 2 printing lora_rx with an RSSI/SNR reading) waits for the second module.

12

Troubleshooting

SymptomTry this
No COM port appears at allUnplug, hold the small BOOT button on the XIAO, plug back in while still holding, release after ~2 seconds. This forces bootloader mode.
pio run -t upload hangs or times outRepeat the BOOT-button trick above, then retry immediately. Also try a different USB port directly on the laptop rather than through a hub.
Upload succeeds but nothing prints in the monitorConfirm the monitor is pointed at the same COM port; close any other program (Arduino IDE, another terminal) that might be holding the port open.
Boot line shows "ina3221":falseCheck the I2C wiring: SDA on D4, SCL on D5, INA3221 address 0x40 (A0 pad tied to GND).
Boot line shows "lora":falseCheck the SPI wiring: CS on D3, DIO0 on D1, RST on D2, SCK on D8, MISO on D0 (GPIO2 — not D9), MOSI on D10.
python not recognizedReopen PowerShell after installing Python; if it still fails, rerun the installer and tick "Add to PATH".
pip install fails with a permissions errorAdd --user: pip install --user platformio pyserial
Pin map reference: the full table lives in firmware/aux/src/main.cpp and firmware/aux/README.md if you need to re-check a solder joint against the source of truth.
