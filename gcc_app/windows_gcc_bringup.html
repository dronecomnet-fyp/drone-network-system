Ground Control Center on Windows
Connect to Node DRONE_A

Build the operator desktop app on the Windows laptop, point it at your live node, and run the two headline operator actions: watch real node health, and issue a personnel PIN.

App gcc_app (Flutter, Windows target) Talks to DRONE_A at https://10.42.0.1:8443 Needs Flutter SDK + Visual Studio C++
Heads up on effort. This is the heaviest install so far: the Flutter SDK plus Visual Studio's C++ tools are a multi-GB download. Stage 0 has a 5-minute check you can do with just a phone and the Pi — do that first to confirm the node handles real traffic before you commit to the toolchain. There is also a lighter alternative (the rescue phone app) noted at the end if you would rather not build the desktop app yet.

00

5-minute sanity check — no install needed

Prove the node handles a real victim message before you build anything. You need a phone and your SSH session to the Pi.

  1. On the phone, join the open WiFi network RESCUE_A. The captive portal ("Send Emergency Message") should pop; if not, open http://10.42.0.1/ in the browser.
  2. Type a message and send it.
  3. On the Pi (over SSH), read it back with the break-glass HQ key:
Pi shellPi · SSH
$ HQ=$(grep HQ_API_KEY ~/rescue-mesh/deploy/secrets/fleet_secrets.env | cut -d= -f2)
$ curl -sk -H "X-API-Key: $HQ" https://10.42.0.1:8443/messages | python3 -m json.tool
Expected
[
  {
    "msg_id": "...",
    "content": "your test message",
    "status": "NEW",
    "node_id": "DRONE_A",
    "time_source": "gps",
    ...
  }
]

Seeing your message here means the victim plane, the database, and the authenticated API all work on real hardware. Now the GCC gives you a proper UI over the same data.

01

Copy the trust files off the Pi

The GCC needs two things from the Pi's deploy/secrets/: the fleet CA certificate (so it can verify the node and refuse imposters), and the break-glass HQ key (your first HQ login, before any PIN exists).

From Windows PowerShell, while the laptop is still on your normal network (the Pi reachable over Ethernet):

PowerShellWindows
PS> scp drone@drone-a.local:~/rescue-mesh/deploy/secrets/fleet_ca.crt $HOME\Downloads\fleet_ca.crt
PS> ssh drone@drone-a.local "grep HQ_API_KEY ~/rescue-mesh/deploy/secrets/fleet_secrets.env"
Expected
fleet_ca.crt                       100%  ...
HQ_API_KEY=hk_1a2b3c4d5e6f...

Save fleet_ca.crt where you can find it (Downloads is fine). Copy the hk_... value somewhere private — it is an admin credential, treat it like a password.

02

Install Flutter and Visual Studio

Two installs, both one-time:

  1. Flutter SDK: follow the official Windows desktop guide. In short: download the Flutter SDK zip, extract it (e.g. to C:\flutter), and add C:\flutter\bin to your PATH.
  2. Visual Studio 2022 Community (free): in the installer, tick the "Desktop development with C++" workload. This is the large download, and it is what compiles the Windows app shell. The lighter "Build Tools" edition also works if you prefer.
Reopen PowerShell after editing PATH so flutter is found in a fresh window.

PS> cd drone-network-system\gcc_app

(a desktop window opens: "Rescue Mesh GCC", a left nav rail with
 Map / Live Feed / Nodes / Personnel / Announcements / Drone / Settings)

The first build is the slow one (it compiles the native shell). Leave flutter run running — the app window is live; closing the terminal closes the app.

If the build fails: I built and tested this app's logic, but the Windows native build has not been run on hardware yet — a plugin might need a small tweak. Paste the error and I will fix it in the repo and push; then git pull and rebuild.
05

Join RESCUE_A and configure the app

Now switch the laptop's WiFi to the node. In Windows WiFi settings, join RESCUE_A (open network, no password). The laptop loses internet while on it — that is expected at a deployment.

If your laptop has Ethernet plugged in for internet, you can stay on both: Ethernet for internet, WiFi on RESCUE_A for the node. Otherwise just join RESCUE_A; you do not need internet for the app to work.

In the GCC, open the Settings tab (bottom of the nav rail) and set:

Settings tab
Node base URL
leave as https://10.42.0.1:8443
Load CA
click it, pick the fleet_ca.crt you copied to Downloads
HQ API key
paste the hk_... value (this is your break-glass HQ login)
Save all
click it

Leave the "Accept ANY certificate (INSECURE)" switch OFF — with the fleet CA loaded, real verification is exactly what you want.

Stop here

Confirm the Nodes tab shows DRONE_A as connected with real GPS and battery numbers matching what /health reported on the Pi. If the badge stays red or you see a TLS error, that is almost always the CA or the WiFi — check the troubleshooting table and tell me what the app shows.

07

Issue a personnel PIN

This is the headline operator action and what unlocks the phone apps. Open the Personnel tab (available because the HQ break-glass key is set).

  1. Click Issue credentials.
  2. Enter a Name (e.g. your own), leave Role as Rescue team, pick an expiry (or none), and Create.
  3. A dialog shows a large 6-digit PIN and a personnel ID like R-014.
What just happened
Personnel ID: R-014
PIN: 4 8 2 9 1 7   (shown once)

The PIN is shown only this once and is never stored in readable form. Write it down — you will use it to log into the rescue phone app in the next runbook. The record now lives on DRONE_A and would sync to other nodes automatically once you have more than one.

Stop here — this is the finish line for the GCC

Confirm you got a personnel ID and a one-time PIN. That proves the whole authenticated plane: the GCC issued a credential, the node hashed and signed it, and it is ready for a rescuer to log in with. Keep the ID + PIN for the rescue-app runbook.

08

Announcement and map (bonus)

Two quick extras to see more of the cockpit:

  • Announcements tab → Compose: publish a short HQ broadcast. It is stored on the node now and will appear in the rescue phone app once that is set up.
  • Map tab: it opens with a "no offline map loaded" banner — that is expected. The offline map needs a region file you prepare before a mission (see docs/OFFLINE_MAPS.md in the repo). Your test message's GPS pin will still render on the blank grid if it had a location attached. Loading a real map is optional and can wait.

Troubleshooting

SymptomTry this
Connection badge stays red / Nodes emptyConfirm the laptop's WiFi is actually on RESCUE_A, not your home network. Then confirm the base URL is https://10.42.0.1:8443.
App shows a TLS / certificate errorThe fleet CA is not loaded or is the wrong file. Re-do Settings → Load CA with the fleet_ca.crt from the Pi. Do NOT flip the insecure switch to work around it.
Personnel tab says "needs an HQ login"The HQ API key field is empty or wrong. Paste the hk_... value from Stage 1 into Settings → HQ API key and Save all.
flutter run fails to compilePaste me the error. Most likely a plugin needs a Windows tweak; it is a repo fix + git pull, not something you debug by hand.
RESCUE_A not in the WiFi listIt is 5 GHz; a few older laptop adapters only see 2.4 GHz. If so, use the rescue phone app (lighter alternative below) from a 5 GHz phone instead.
git clone asks for a passwordThe repo is private. Use a GitHub personal access token as the password, or tell me and I can make the repo public.
Lighter alternative: if the desktop toolchain is too much right now, the rescue phone app reaches the same node and the same PIN login from an Android phone — that is the next runbook, and it needs no Visual Studio. Issue the PIN here (or via the Stage 0 curl with the HQ key) and you are ready for it.
